The layers from the last lesson aren't static — data moves through them constantly. Being able to narrate that round trip, hop by hop, is the single most common "do you understand hardware?" test.

## The journey, one hop at a time

<DataFlow caption="Data flows up to the cloud and out to the app; commands and updates flow back down the same path." />

Take a connected parcel locker as the example:

1. **A parcel is dropped in.** A **sensor** on the device detects it.
2. **The firmware logs the event.** The **MCU** running the firmware records what happened. *(Owner: firmware/embedded.)*
3. **The device sends it out.** Over its connection — Wi-Fi, cellular, etc. — typically using a lightweight protocol like **MQTT**. *(Owner: firmware + connectivity.)*
4. **The cloud updates its state.** The backend records "locker 12 now has a parcel." *(Owner: cloud.)*
5. **The user is notified.** The cloud sends a push notification; the courier's **app** fetches the new state via an **API** and shows it. *(Owner: cloud + app.)*

The whole trip takes seconds — but it crossed four teams.

## It flows back down, too

The same path runs in reverse for **commands** and **updates**:

- A user (or the cloud) sends a **command** — "unlock locker 12" — down to the device.
- The cloud pushes an **OTA** firmware update to the **fleet**; each device validates it, flashes it, and reports success or rolls back.

## What this matters for a PM

- **Triage by ownership.** When something's wrong end-to-end, the flow tells you *which hop and which team* to start with — instead of everyone pointing at each other.
- **The hard trade-offs live on the way up.** How the device talks to the cloud (Wi-Fi vs cellular vs BLE) trades range, power and recurring cost against each other — a core PM decision, covered in the Connectivity topic.
- **Updates need care.** Because you can't recall physical units, OTA rollouts are staged and reversible. Your software-release instincts apply — just with higher stakes.
